# CampusHelpATW MVP 功能清单（P0/P1/P2）

## 1. 文档目的
用于将《项目思想》转化为可执行开发任务。

- `P0`：MVP 必须实现，缺失会导致核心链路不可用。
- `P1`：建议在 MVP 后尽快补齐，明显提升体验和稳定性。
- `P2`：中期增强项，不影响初期验证核心价值。

## 2. MVP 范围定义
MVP 只验证一件事：
校园用户能否在平台内完成“发布求助 -> 接单协作 -> 完成确认 -> 评价结算”的闭环。

## 3. P0 功能清单（必须）

### 3.1 账号与身份
1. 校园邮箱注册与登录（邮箱验证码）。
2. 登录态管理（token + 过期处理）。
3. 用户资料基础页（昵称、头像、院系可选）。
4. 用户信誉信息展示（帮助点、荣誉点、完成单数）。

验收标准：
1. 未认证用户无法发布和接取任务。
2. token 失效后可被正确识别并要求重新登录。
3. 用户可查看自己的基础信誉信息。

### 3.2 任务系统（核心闭环）
1. 发布任务（标题、描述、类别、地点、截止时间、报酬）。
2. 任务列表浏览与筛选（类别、最新、报酬）。
3. 接单、取消、完成确认。
4. 任务状态机：
`OPEN -> ACCEPTED -> IN_PROGRESS -> PENDING_CONFIRM -> DONE/CANCELED/DISPUTED`

验收标准：
1. 一条任务同一时间只允许一个有效帮助者（MVP 简化）。
2. 状态变更必须符合状态机，不允许跳转。
3. 所有状态变更均可追溯（有日志记录）。

### 3.3 沟通与通知
1. 任务内双人聊天（仅发布者和接单者可见）。
2. 关键事件通知（接单、取消、完成、评价待处理）。

验收标准：
1. 非任务参与者无法读取聊天内容。
2. 关键节点触发后 3 秒内可看到通知（站内）。

### 3.4 评价与积分
1. 双方互评（星级 + 文本）。
2. 完成任务后结算帮助点与荣誉点。
3. 任务发布消耗帮助点（避免滥发）。

验收标准：
1. 未完成任务不可评价。
2. 每条任务每人仅可评价一次。
3. 积分变更必须有流水记录。

### 3.5 治理与风控（基础）
1. 举报功能（任务、帖子、聊天消息）。
2. 后台审核入口（管理员可处理举报）。
3. 基础限流（单位时间发帖/发任务上限）。

验收标准：
1. 举报后进入待处理队列。
2. 管理员处理后可追踪处理结果。
3. 高频恶意操作可被拦截。

## 4. P1 功能清单（推荐）

1. 事务协作体验增强（筛选、归档检索、消息摘要、沟通效率提升）。
2. 论坛模块完善（帖子标签、热度排序）。
3. 任务超时自动流转（超时提醒和自动关闭规则）。
4. 低信誉用户功能限制（接单频控、发单门槛）。
5. 积分兑换页（帮助点 -> 平台奖励）。
6. 分类与系统参数后台配置（任务分类、积分倍率、风控阈值）。

验收标准：
1. 事务协作入口稳定，归档后只读规则与未读同步一致。
2. 超时任务按规则自动处理并通知相关用户。
3. 低信誉限制可配置、可灰度调整。

## 5. P2 功能清单（增强）

1. 个性化任务推荐。
2. 任务匹配质量评分。
3. 校园组织官方任务池。
4. 跨校认证与多学校切换。
5. 活动运营系统（任务挑战、周榜）。

## 6. 非功能要求（MVP 阶段最低标准）

1. 性能：核心接口 P95 < 500ms（本地环境不计网络抖动）。
2. 稳定性：核心任务链路可用性目标 >= 99%（内测期按天统计）。
3. 安全：密码/验证码不明文存储，关键接口需鉴权。
4. 审计：任务状态变更、积分变更、管理员操作需留痕。

## 7. 推荐开发顺序（两周一迭代）

1. 迭代 1：账号认证 + 任务发布/列表 + 基础状态流转。
2. 迭代 2：接单协作 + 聊天 + 完成确认。
3. 迭代 3：评价积分 + 举报审核 + 内测修复。

## 8. 风险与应对

1. 风险：恶意刷分。
应对：单日积分上限 + 行为风控 + 异常回滚机制。

2. 风险：任务纠纷处理成本高。
应对：保留关键沟通日志，先规则化仲裁，再人工介入。

3. 风险：冷启动供需不平衡。
应对：先聚焦 1-2 个高频场景（跑腿、资料代取）做密度。
