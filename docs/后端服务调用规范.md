# CampusHelpATW 后端服务调用规范（MVP）

## 1. 目标
统一服务层与数据库层的职责边界，避免以下问题：
1. 任务状态与日志不一致。
2. 积分余额与积分流水不一致。
3. 举报处理无审计记录。

## 2. 总体原则
1. 任务状态变更必须走 `sp_task_transition`。
2. 积分余额变更必须走 `sp_add_points`。
3. 管理端处理动作必须写 `admin_operation_logs`。
4. 业务层保留同等校验，触发器/存储过程只做兜底。

## 3. 模块调用约定

### 3.1 任务模块
接口与过程映射：
1. `POST /tasks/{id}/accept`
   - 检查：任务存在、状态为 `OPEN`、操作者非发布者。
   - 更新：先写 `acceptor_id`，后调 `sp_task_transition(id, 'ACCEPTED', operator, reason)`。

2. `POST /tasks/{id}/start`
   - 检查：状态 `ACCEPTED` 且操作者为接单者。
   - 更新：调 `sp_task_transition(id, 'IN_PROGRESS', operator, reason)`。

3. `POST /tasks/{id}/submit-completion`
   - 检查：状态 `IN_PROGRESS` 且操作者为接单者。
   - 更新：调 `sp_task_transition(id, 'PENDING_CONFIRM', operator, reason)`。

4. `POST /tasks/{id}/confirm-completion`
   - 检查：状态 `PENDING_CONFIRM` 且操作者为发布者。
   - 更新：
     - 调 `sp_task_transition(id, 'DONE', operator, reason)`。
     - 再按规则调用 `sp_add_points`（见 3.2）。

5. `POST /tasks/{id}/dispute`
   - 检查：状态在可申诉范围（建议 `IN_PROGRESS`/`PENDING_CONFIRM`）。
   - 更新：调 `sp_task_transition(id, 'DISPUTED', operator, reason)`。

### 3.2 积分模块
1. 发布任务消耗帮助点：
   - `sp_add_points(user_id, 'HELP', -cost, 'TASK_PUBLISH', task_id, remark)`
2. 接单完成奖励帮助者：
   - `sp_add_points(acceptor_id, 'HELP', +x, 'TASK_COMPLETE', task_id, remark)`
   - `sp_add_points(acceptor_id, 'HONOR', +y, 'TASK_COMPLETE', task_id, remark)`
3. 发布者确认奖励：
   - `sp_add_points(publisher_id, 'HONOR', +z, 'TASK_CONFIRM', task_id, remark)`

禁止：
1. 直接 `UPDATE users SET help_points_balance = ...`。
2. 只写余额不写 `point_ledger`。

### 3.3 举报与仲裁模块
1. 用户举报：
   - 写 `reports`，状态 `PENDING`。
2. 管理员处理举报：
   - 更新 `reports.status` 与 `handled_at`。
   - 同时写 `admin_operation_logs`。
3. 管理员仲裁任务：
   - 按决策调用 `sp_task_transition` 到 `DONE` 或 `CANCELED`。
   - 如涉及积分补偿，调用 `sp_add_points`。

## 4. 事务边界建议
1. 过程内已事务化：
   - `sp_task_transition`
   - `sp_add_points`
2. 服务层建议：
   - 不要在外层再包裹大事务去调用上述过程。
   - 避免“外层事务 + 内层过程事务”导致行为不可预期。
3. 多步骤动作（如确认完成 + 积分发放）：
   - 允许按步骤调用过程。
   - 任一步失败应返回业务错误，并记录告警日志。

## 5. 幂等与重复提交
1. 关键接口使用 `Idempotency-Key`（建议 header）。
2. 在 DB 侧增加业务去重策略：
   - 评价：`unique(task_id, reviewer_id)`。
   - 点赞：`unique(post_id, user_id)`。
   - 积分：按 `user_id + point_type + biz_type + biz_id` 做幂等检查。

## 6. 错误码映射建议
1. SQLSTATE `45000` 且含 `invalid task status transition` -> `4009`。
2. SQLSTATE `45000` 且含 `insufficient ... points` -> `4009`。
3. SQLSTATE `45000` 且含 `user not found`/`task not found` -> `4004`。
4. 其他 DB 异常 -> `5000`。

## 7. 审计与日志要求
1. 每次任务状态变更都应有：
   - 接口访问日志
   - `task_status_logs`
2. 每次积分变化都应有：
   - 业务日志（含 `biz_type/biz_id`）
   - `point_ledger`
3. 每次管理动作都应有：
   - `admin_operation_logs`
   - 操作者 ID、对象 ID、处理结论

## 8. 代码组织建议
1. `TaskService`：只负责参数校验、权限判断、调用过程。
2. `PointsService`：统一封装 `sp_add_points`。
3. `ModerationService`：举报与仲裁处理。
4. `Repository` 层禁止直接改积分余额和任务状态（除创建任务初始状态）。

## 9. 最小联调断言（后端自测）
1. `accept -> start -> submit-completion -> confirm-completion` 全链路通过。
2. 状态日志条数与变更次数一致。
3. 完成任务后积分余额与流水可对账。
4. 非法跳转（例如 `OPEN -> DONE`）被拒绝。
